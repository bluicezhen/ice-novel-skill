# 冰小说创作技能 — 需求文档

> 版本：v0.2 draft
> 日期：2026-05-25

---

## 一、核心理念

辅助中文网文长篇创作的 Claude Code Skill。核心设计目标：

1. **低上下文消耗** — 通过 Obsidian 风格文件链接（`[[filename]]`）按需加载，避免读入无关信息
2. **人工可控** — 逐幕生成、逐幕校对，AI 不擅自推进
3. **一致性保障** — 角色弧光规划 + 三维演变追踪 + 伏笔追踪 + 三级检查体系

---

## 二、工作流总览

```
提出 idea
  ↓
确定主线故事 + 世界观设定 + 主要事物设定 + 主要人物小传
  ↓
规划角色弧光 + 编制伏笔表
  ↓
拆分为 3～10 卷 → 每卷拆为 3～10 篇 → 每篇拆为 3～10 章
  ↓
确定风格（填写 style-guide.md）← 必须在开始写之前完成
  ↓
/写章：AI 生成章前简报 → 用户确认
  ↓
逐幕生成（AI 写 1 幕 → 人工校对 → 写下一幕）
  ↓
章级检查 → 篇级检查 → 卷级检查
```

### 各阶段产出物

| 阶段 | 产出物 | 人工介入点 |
|------|--------|-----------|
| 1. 创意孵化 | `idea.md` | 确认核心创意 |
| 2. 世界观构建 | `world.md`, `characters/*.md`, `settings/*.md` | 逐项审核设定 |
| 3. 大纲拆解 + 弧光与伏笔 | `outline.md`（含角色弧光规划 + 伏笔规划总表） | 确认结构与角色走向 |
| 4. 风格确定 | `style-guide.md`（含正面示范 + 禁止清单） | 确认风格锚点和场景类型标杆 |
| 5. 章节写作 | `draft/volX/partY/chZ.md`（每章 3～5 幕） | 每幕校对 |
| 6. 层级检查 | `reports/` 下各级检查报告 | 确认检查结论 |

## 四、大纲结构（outline.md）

`outline.md` 是整个写作系统的脊梁骨，不只是目录树。它需要同时满足三个读者：

| 读者 | 需求 |
|------|------|
| **作者** | 可读可改、一眼看清故事全貌 |
| **AI 写作时** | 从中提取章前简报，知道这一章写什么、边界在哪 |
| **AI 检查时** | 对照大纲判断剧情是否越界、节奏是否偏离 |

### 核心原则

1. 每章必须有可执行的剧情摘要（2-4 句），不能只写标题
2. 每章必须标注禁区——什么还不能写，这是防止 AI 剧情加速的关键机制
3. 角色弧光在大纲阶段就规划好，不能写完再追踪
4. 用 `[[]]` 链接角色和设定

### 大纲文件结构

```
outline.md
├── 1. 故事概要         （一段话讲清这本书）
├── 2. 核心冲突         （主线矛盾是什么）
├── 3. 角色弧光规划      （每个核心角色的起点→转折→终点）
├── 4. 卷纲             （每卷一句话 + 篇→章→章级表格）
└── 5. 伏笔规划总表      （所有伏笔汇总，卷纲各章引用 ID）
```

### 4.1 故事概要

```markdown
# 故事概要

> 一句话版：_______________________________
>
> 一段话版（约 200 字）：
> _______________________________
```

### 4.2 核心冲突

```markdown
# 核心冲突

- **主线矛盾**：_______________________________（谁 vs 谁，争什么）
- **内在冲突**：_______________________________（主角内心的核心矛盾）
- **最终目标**：_______________________________（主角最终要达成什么）
- **主要阻碍**：_______________________________（最大的障碍是什么）
```

### 4.3 角色弧光规划

在大纲阶段就把每个核心角色的演变路径定好。后续 `characters/*.md` 追踪实际执行情况，与规划对照。

```markdown
# 角色弧光规划

## [[protagonist]] — 主角

- **弧光类型**：[成长型 / 堕落型 / 救赎型 / 平弧型]
- **弧光概述**：从（初始状态）→ 经过（核心转折）→ 达到（最终状态）

**三条线里程碑**（每列填具体章节号和触发事件）：

| 维度 | ch01 起点 | 里程碑1（chX） | 里程碑2（chX） | 里程碑3（chX） | 终点（chX） |
|------|----------|---------------|---------------|---------------|------------|
| 能力 | 练气初期 | ch15 突破练气中期（宗门大比压力下突破） | ch30 获得传承（通过秘境试炼） | — | ch50 元婴初期 |
| 性格 | 怯懦逃避 | ch10 开始承担责任（师父之死逼迫） | ch25 主动布局（为保护同伴设局） | — | ch50 沉稳果决 |
| 心态 | 只想活着回家 | ch20 认同这个世界（第一个朋友舍命相救） | ch40 有了要保护的人 | — | ch50 守护此界 |

> 不要求三条线同步变化——可能能力线变了但心态还没跟上，这正是弧光的张力所在。

## [[antagonist]] — 反派
（同上格式）

## [[重要配角]]
（同上格式，只列有弧光的核心角色）
```

**填写规则**：

| 规则 | 说明 |
|------|------|
| 每列必须有具体章号 | 不能写"中段"或"后期"，必须精确到 chXX |
| 每列必须有触发事件 | 不能只写"变强了"，要写"什么事件导致变强" |
| 里程碑数量不做硬性要求 | 没有变化的维度可填"—"，但至少有一个维度有 2+ 里程碑 |
| 与章级大纲对齐 | 弧光表中的 chXX 必须在对应章的大纲表格中体现，不能弧光说 ch20 突破但大纲 ch20 没写 |

### 4.4 卷纲（主体）

每章一个结构化表格，AI 写作和检查时频繁参考。

```markdown
# 卷纲

## 第一卷：<卷名>（第 1～N 章）

> **卷概要**：一句话说明本卷讲什么。
> **卷级情感曲线**：_______________________________
> **本卷结束时读者应该感到**：_______________________________

---

### 第一篇：<篇名>（第 1～N 章）

> **篇概要**：这篇讲什么，结束时到达什么状态。
> **篇级弧光**：这篇推进了谁的什么弧光。

#### 第 1 章：<章名>

| 字段 | 内容 |
|------|------|
| **剧情摘要** | （2-4 句，说清本章发生什么、为什么发生、结果是什么） |
| **节奏档位** | 慢档 / 中档 / 快档 |
| **视角角色** | [[protagonist]] |
| **出场角色** | [[protagonist]]、[[mentor]] |
| **本章目标** | （本章必须完成的事——角色走到了哪一步） |
| **弧光推进** | （本章推动哪个角色的哪条线：能力/性格/心态，触发事件是什么。无则填"—"） |
| **章末钩子** | （本章结束时读者产生的悬念/期待是什么） |
| **禁区** | （本章绝对不能写的内容：不能揭露的秘密、不能超出的能力边界等） |
| **埋设伏笔** | F0XX、F0XX |
| **回收伏笔** | F0XX |
| **情绪走向** | 从（情绪A）→ 到（情绪B） |
| **关联章节** | 承接 ch(N-1) 的 XX 线 / 为 ch(N+2) 的 XX 做铺垫 |

#### 第 2 章：<章名>
（同上格式）
```

**各字段的意义**：

| 字段 | 为什么重要 | 不填的后果 |
|------|-----------|-----------|
| **剧情摘要** | AI 写章时最重要的输入 | AI 自由发挥，剧情漂移 |
| **弧光推进** | 告诉 AI 本章重点刻画角色的哪条线、触发什么变化 | AI 写的每个场景力道均等，该突出的写淡了 |
| **禁区** | 防止 AI 提前揭露关键信息 | 爽点提前泄气，节奏崩盘 |
| **章末钩子** | 约束每章结尾必须留悬念 | 章节平平收尾，读者流失 |
| **情绪走向** | 约束每章的情感弧线 | 章节情绪扁平，人物没变化 |
| **节奏档位** | 卷级节奏控制 | 连续高潮或连续慢章拖死读者 |
| **关联章节** | 跨章写作时 AI 知道前后衔接 | 章节之间割裂 |

### 4.5 伏笔规划总表

所有伏笔在开写前定好 ID，卷纲各章通过 ID 引用。

```markdown
# 伏笔规划总表

| ID | 伏笔内容 | 埋设章 | 回收章 | 关联角色 | 类型 | 备注 |
|----|---------|--------|--------|---------|------|------|
| F001 | 主角手臂印记第一次发光 | ch03 | ch20 | [[protagonist]] | 能力伏笔 | |
| F002 | 师父留下的锦囊 | ch08 | ch15 | [[mentor]] | 道具伏笔 | ch12 给一次暗示 |
```

伏笔类型：能力伏笔、道具伏笔、信息伏笔、人物伏笔。

注意：此表是**规划表**（静态），与 `foreshadow.md`（运行时状态追踪，动态）互补——前者定义什么伏笔在什么章埋收，后者追踪当前执行到哪一步。

### 4.6 完整示例

```markdown
## 第一卷：初入江湖（第 1～30 章）

> **卷概要**：主角穿越到修真世界，在底层挣扎求生，逐渐发现这具身体的原主人留下了一个巨大的秘密。
> **卷级情感曲线**：慢档起步→渐进升温→中段小高潮→缓→卷末爆发
> **本卷结束时读者应该感到**：主角终于有了立足之地，但更大的威胁刚刚露出冰山一角。

---

### 第一篇：挣扎求生（第 1～10 章）

> **篇概要**：主角适应新身份，在底层修真坊市站稳脚跟，意外获得一件神秘物品。
> **篇级弧光**：主角从被动适应→开始主动求生。

#### 第 1 章：醒来

| 字段 | 内容 |
|------|------|
| **剧情摘要** | 主角从昏迷中醒来，发现自己穿越到一个被打伤的练气期修士身上。他整理了残存的记忆碎片，发现自己处境危险——原主人在被追杀。他拖着伤体在坊市角落找到一处藏身之所。 |
| **节奏档位** | 慢档 |
| **视角角色** | [[protagonist]] |
| **出场角色** | [[protagonist]] |
| **本章目标** | 建立穿越事实；展示世界第一印象；让读者感受到危机感 |
| **弧光推进** | —（第一章为初始状态，不做变化） |
| **章末钩子** | 主角在藏身处翻看原主人的储物袋，里面有一封信——署名者三天后会来取一件东西，如果拿不到，"你知道后果"。 |
| **禁区** | 不能揭示追杀者是谁；不能展示力量体系细节（第二章才展开） |
| **埋设伏笔** | F001（储物袋中的信） |
| **回收伏笔** | — |
| **情绪走向** | 困惑恐慌 → 冷静观察 → 找到藏身处后暂松一口气 → 看信后再度紧张 |
| **关联章节** | 为 ch02 的坊市探索和力量体系展示做铺垫 |
```

## 五、Obsidian 链接机制

### 链接语法

所有文件间引用使用 `[[filename]]` 语法（不含 `.md` 扩展名，不含路径目录名）。文件名在项目内唯一，AI 通过文件名定位。

```markdown
# world.md
## 核心人物
- 主角：[[protagonist]]
- 反派：[[antagonist]]

## 地理
详见 [[geography]]
```

### 默认加载规则

AI 在生成章前简报时，按以下规则解析链接并生成文件加载清单：

1. **起点**：outline 当前章表格（含出场角色、弧光推进、禁区、伏笔等字段中的 `[[链接]]`）
2. **第 1 层**：起点中所有直接链接的文件（如角色的 `characters/*.md`、设定的 `settings/*.md`）
3. **第 2 层**：第 1 层文件中出现的 `[[链接]]`（如角色文件中引用的设定、其他角色）
4. **停止**：第 3 层及更深的不再追踪

```
起点：outline 当前章表格
  ├─ [[protagonist]]           ← 第1层，加载
  │   └─ [[power-system]]      ← 第2层，加载
  │       └─ [[ancient-ruins]]  ← 第3层，不加载
  ├─ [[mentor]]                ← 第1层，加载
  │   └─ [[factions]]          ← 第2层，加载
  └─ [[geography]]             ← 第1层，加载
```

### 作者手动调整

简报中「必须加载的文件」清单生成后，作者可以：

- **追加**：某个第 3 层文件很重要，手动加入清单
- **删除**：某个第 1/2 层文件与本章无关，手动移除
- **替换**：觉得 AI 选的文件不对，替换为正确的文件

作者的调整直接生效，跳过自动规则。简报确认后 AI 按调整后的清单加载。

### 为什么默认 2 层

- 1 层太少：只看角色文件本体，看不到角色关联的设定
- 3 层太深：容易出现链路爆炸（一个设定链接另一个设定→链接另一个设定→……）
- 2 层刚好：角色 + 角色的直接关联设定 → 覆盖本章需要的信息，深度失控时作者手动追加
- 作者覆盖权兜底：2 层是一个安全的默认值，不是硬限制

### 反向链接维护

`[[链接]]` 是单向的——从 A 指向 B，无法从 B 知道谁指向了自己。因此每个文件末尾维护一段反向链接，AI 自动同步。

**格式**（每个文件尾部）：

```markdown
## 反向链接
<!-- AI 自动维护。创建/修改/删除 [[链接]] 时自动更新目标文件的此段。 -->
- [[protagonist]] — 能力演变表引用
- [[antagonist]] — 力量来源引用
```

**维护规则**：

| 触发时机 | AI 做什么 |
|---------|---------|
| 在任何文件中**新增** `[[X]]` | 在 X 的反向链接段追加一条（注明来源文件和引用原因） |
| 在任何文件中**删除** `[[X]]` | 在 X 的反向链接段移除对应条目 |
| 执行 `/拆大纲` | 全量扫描所有文件，重建全部反向链接（首次初始化） |
| 执行 `/更新设定` | 修改设定文件后，用反向链接定位所有引用者，逐条检查是否需要同步 |

**为什么不全靠全文搜索**：`grep` 搜 `[[power-system]]` 能找到精确匹配，但无法区分"能力演变表引用"和"随口提了一句"——反向链接带原因标注，级联更新时可以优先检查真正的依赖关系。

## 六、风格指南（style-guide.md）

### 定位

`style-guide.md` 的目标是让 AI 知道"要写成什么样"，不只是"不能写成什么样"。因此在开写前必须完成，不迟于大纲拆解后、第一章写作前。

### 禁止清单来源：预设 + 积累

- **预设**：skill 内置通用 AI 写作禁止清单，开书时写入
- **积累**：每次章级检查发现新风格问题，自动追加
- 用户可随时手动编辑

### 文件结构

```markdown
# 风格指南

## 一、文风 DNA（定性描述，开书时填写）

- **叙述者态度**：[冷眼旁观 / 投入共情 / 调侃毒舌 / 威严庄重 / ...]
- **幽默类型**：[无 / 冷笑话 / 人物吐槽 / 荒诞对比 / ...]
- **情感描写方式**：[白描行动不写内心 / 大量内心独白 / 通过他人反应间接写 / ...]
- **叙事节奏偏好**：[快节奏少描写 / 张弛交替 / 细腻铺陈 / ...]

## 二、叙事参数（开书时填写）

- **人称**：[第一人称 / 第三人称有限 / 第三人称全知]
- **叙事距离**：[贴近（大量内心独白） / 中等 / 远观（少写内心）]
- **时态**：[过去式 / 现在式]
- **平均句长**：约 XX 字
- **段落长度**：约 X～X 行
- **对话占比**：约 XX%

## 三、对标作品（可选但建议填）

- **参考作品**：_______________________________
- **可学的特征**：（人话版，如"对话从不解释，读者自己猜人物在想什么""打斗描写极简，重在战前气氛"）

> 有对标作品时，AI 可以据此校准"味道"；没有也不影响。

## 四、正面示范（必填）

### 风格锚点（至少 3 段）

每段包含原文 + 为什么选 + AI 可学什么。不需要完整场景，50～200 字即可。

#### 锚点1：<场景类型>

```
<原文段落>
```

- **为什么选这段**：（1-2 句：体现了什么风格特征）

### 场景类型标杆

不同场景类型的写法特征不同，AI 需要知道区别：

| 场景类型 | 标杆特征 | 参考样本 |
|---------|---------|---------|
| 动作/战斗 | 短句为主，动词密集，少形容词，每个动作有因果 | 见锚点 N |
| 日常/对话 | 对话占比高，潜台词 > 明说，动作穿插打断对话节奏 | 见锚点 N |
| 内心独白 | 贴近角色认知水平，不做上帝视角总结，允许碎片化 | 见锚点 N |
| 环境/氛围 | 用角色感官过滤环境（角色注意到什么 = 角色在乎什么），不架空描写 | 见锚点 N |
| 高潮/爆点 | 节奏最快，情绪最浓，释放之前压抑的张力 | 见锚点 N |

> 不要求填满所有 5 种类型。最少填 3 种 — 最常写的场景类型优先。

## 六、禁止清单

### 预设禁止句式
| 可用频次 | 句式 | 替代方案 |
|---------|------|---------|
| 禁止 | "不禁XXX" | 直接写行动 |
| 禁止 | "映入眼帘" | 直接写看到了什么 |
| 禁止 | "只见/此时此刻" | 直接切换场景 |
| ≤1次/章 | "仿佛/宛如/宛若" | 用具体感知替代 |
| ≤2次/章 | "心中暗道/暗自思忖" | 改为行动或删掉 |
| ≤2次/章 | "沉声道/淡淡地说" | 统一用"说"或删标签 |

### 预设禁止词汇
| 禁止词 | 替代方案 | 原因 |
|--------|---------|------|
| 踱步 | 走来走去 | AI高频词 |
| 不由得/不由自主/情不自禁 | 删除或改写 | 主体性剥夺 |
| 嘴角微扬/勾起一抹弧度 | "他笑了" | AI特征极强 |
| 目光如炬/目光深邃 | 写眼睛看向哪里 | 空洞套话 |
| 脸色一变/身形一顿 | 写具体生理反应 | 反应套话 |
| 未来可期/前途无量 | 用具体后续影响替代 | 空洞结语 |

### 积累禁止项（写作中追加）
| 发现章节 | 禁止项 | 替代方案 | 原因 |
|---------|--------|---------|------|
```

### 为什么正面示范要带注解

光给 AI 范文没有用——它只能表面模仿句式词汇，不理解**为什么**这是标杆。"原文 + 为什么选 + AI 可学什么"的三段结构让 AI 知道模仿的方向，而不仅仅是模仿的对象。

### 为什么场景类型要分表

写动作戏和写日常对话的写法完全不同。如果只给一个笼统的"风格描述"，AI 写什么场景都一个味道。场景类型标杆让 AI 知道：切到打斗时收紧句子，切到对话时放松节奏。

### 为什么用定性描述替代量化参数

"对话占比 40%"远不如"对话从不直说，每句话都藏着另一层意思"有用。前者约束形式，后者约束灵魂。定量参数（句长、段落长度等）保留在叙事参数中作为补充，但不作为风格的核心锚点。

## 十一、章前简报

### 概念

章前简报是每次 `/写章` 时 AI 在开始写第一幕之前，自动生成的一份结构化提示。它告诉 AI：
1. **加载什么**——哪些文件必须读入上下文
2. **写什么**——核心写作指令的浓缩
3. **边界在哪**——本章绝对不能做的事

简报解决了"AI 自己判断就容易漏"的问题——不依赖 AI 在写幕时自行决定读哪些文件。

### 生成时机

`/写章` 的第一步，AI 根据 outline 当前章表格自动生成简报 → 展示给用户 → 用户确认后进入第一幕。每章只生成一次。

### 简报格式

```markdown
# 章前简报 — 第 X 章：<章名>

## 一、必须加载的文件

（AI 按 2 层链接规则自动生成，作者确认前可增删）

| 层级 | 文件 | 加载原因 | 作者调整 |
|------|------|---------|---------|
| 起点 | outline.md 当前章表格 | 本章核心指令来源 | — |
| L1 | [[style-guide]] → 场景类型标杆 | 本章主要场景类型对应的风格参照 | |
| L1 | [[protagonist]] | 本章视角角色 | |
| L1 | [[出场角色B]] | 本章出场 | |
| L2 | [[power-system]]（来自 [[protagonist]]） | 本章涉及力量体系展示 | |
| L2 | [[foreshadow]] → 计划回收项 | 本章需回收 F0XX | |
| + | （作者自行追加） | | ✏️ |
| - | （作者移除不需要的文件） | | ✏️ |

## 二、本章核心指令

- **剧情任务**：（从 outline 剧情摘要提炼）
- **角色任务**：（本章角色要走到哪一步）
- **情绪线**：（从情绪A → 到情绪B，分幕拆解）
- **章末钩子**：（本章结尾必须抵达的悬念）
- **禁区**：（绝对不能写的内容）

## 三、伏笔任务

- **待埋设**：F0XX（内容简述）
- **待回收**：F0XX（回收方式提示）

## 四、承接上文

- **上一章结尾状态**：（角色位置、情绪、正在发生什么）
- **上次章末钩子**：（上一章留下的悬念，本章需要呼应或推进）

## 五、幕结构建议

| 幕 | 剧情功能 | 情绪 | 预估字数 | 与上一幕衔接点 |
|----|---------|------|---------|--------------|
| 第1幕 | 场景建立，角色当前状态 | 情绪A | 约 XXX 字 | — |
| 第2幕 | 冲突展开 | 过渡 | 约 XXX 字 | 承接第1幕末尾留下的问题/决定/危险 |
| 第3幕 | 转折/高潮 | 情绪B | 约 XXX 字 | 承接第2幕的结果或反转前奏 |
| （根据 outline 情绪走向和本章总字数 2000-4000 字拆分 3-5 幕） |
```

### 简报使用方式

1. AI 生成简报后展示给用户，用户可以修改任何字段（尤其是禁区、幕结构建议）
2. 用户确认后，AI 将简报内容作为本次 `/写章` 的上下文锚点
3. 每写一幕时，AI 回顾简报确认当前幕是否符合计划，无需重新扫描全部文件
4. 简报随对话上下文自然存在，不写磁盘文件（除非用户要求）

## 十二、幕的结构定义

### 12.1 字数约束

每章 2000～4000 字。每幕的字数由作者与 AI 协作分配，参考如下：

| 幕数 | 每幕约字数 | 适用场景 |
|------|-----------|---------|
| 3 幕 | 700～1300 字/幕 | 慢档/过渡章 |
| 4 幕 | 500～1000 字/幕 | 中档章节（最常用） |
| 5 幕 | 400～800 字/幕 | 快档/高潮章 |

不要求各幕字数均等——冲突幕可以更长，过渡幕可以更短。

### 12.2 幕的内部结构

每幕是一个连续的叙事单元，包含三个阶段：

| 阶段 | 功能 | 约占幕字数 |
|------|------|-----------|
| **进入** | 建立场景：在哪里、谁在场、当前什么状态 | 10-20% |
| **推进** | 冲突发展或人物刻画：发生了什么、角色做了什么 | 60-70% |
| **落点** | 本幕结束时读者的感受：新的问题/期待/张力 | 10-20% |

落点是幕最重要的部分——每幕结束时读者都应该有一个"想知道接下来怎样"的念头。

### 12.3 幕数选择

| 幕数 | 典型分配 | 节奏特征 |
|------|---------|---------|
| 3 幕 | 铺垫 → 冲突 → 收束 | 缓起缓收，重在角色刻画和氛围 |
| 4 幕 | 铺垫 → 展开 → 转折 → 钩子 | 最常用，标准网文节奏 |
| 5 幕 | 进入 → 铺垫 → 冲突 → 转折 → 钩子 | 多一幕冲突推进，适合高潮章 |

### 12.4 幕间衔接

逐幕生成时，后一幕必须承接前一幕的落点，不能把各幕写成彼此割裂的独立片段。

#### 衔接规则
1. **承接落点**：第 N 幕开头必须回应第 N-1 幕结尾留下的问题、危险、决定、情绪转折或信息揭露。
2. **状态连续**：幕首应保持人物状态、情境状态、悬念状态的连续；若发生变化，需明确交代。
3. **跳转需说明**：若幕间发生时间推进、地点切换、人物显著状态变化，必须有一句过渡说明。
4. **视角稳定**：同一章内默认保持 outline 指定视角；如需切换，必须在章前简报中明确批准。
5. **禁止硬切**：不得无说明跨越关键事件、直接跳过角色应有反应，或让上一幕冲突在下一幕开头无故消失。
6. **最后一幕落点**：最后一幕的落点 = 章末钩子。

#### 幕结构建议中的衔接点
建议在章前简报的幕结构建议表中，额外标注每幕与上一幕的衔接点，例如：

```markdown
| 幕 | 剧情功能 | 情绪 | 预估字数 | 与上一幕衔接点 |
|----|---------|------|---------|--------------|
| 第1幕 | 场景建立 | 压抑 | 约 XXX 字 | — |
| 第2幕 | 冲突展开 | 紧张升高 | 约 XXX 字 | 承接第1幕末尾主角发现的异常动静 |
| 第3幕 | 局势反转 | 惊愕 | 约 XXX 字 | 承接第2幕主角决定主动试探的结果 |
| 第4幕 | 章末钩子 | 危机 | 约 XXX 字 | 承接第3幕揭露的信息，落到更大的威胁 |
```

### 12.5 幕规划流程

生成章前简报时，AI 根据 outline 中本章的剧情摘要和情绪走向，在简报「五、幕结构建议」中建议：

1. 本章用几幕（3/4/5）
2. 每幕的剧情功能和预估字数
3. 幕与幕之间的情绪推进

作者可以调整任何字段，确认后 AI 按此结构逐幕生成。每幕生成后 AI 会告知当前字数，作者可以据此调整后续幕的篇幅。

## 十三、上下文加载策略

简报生成阶段自动完成链接解析和文件加载清单。每次 `/写章` 的实际执行：

```
步骤 1：生成简报
  AI 读取 outline 当前章表格
  → 解析所有 [[链接]]，追踪 2 层生成文件清单
  → 展示清单 + 核心指令 + 幕结构建议 → 等待用户确认
  → 用户增删文件、调整字段后确认

步骤 2：写第 1 幕
  按确认后的文件清单加载 → 按核心指令 + 幕结构建议生成 → 展示

步骤 3：写第 N 幕（N > 1）
  简报（已在上下文）+ 前 N-1 幕正文（已在上下文）
  → 检查当前幕是否偏离简报 → 生成 → 展示

步骤 4：全部幕完成后
  自动执行章级检查
```

加载上限：默认 2 层链接追踪生成的文件清单通常不超过 6 个文件。如超出，AI 提示作者精简；作者也可手动追加超过 2 层的文件。

## 十四、幕级生成交互（逐幕模式）

```
用户：/写章
AI：生成章前简报（含幕结构建议 + 各幕预估字数）→ 展示
用户：确认简报 / 修改幕数和字数分配
AI：加载简报指定文件 → 生成第 1 幕 → 展示（附当前字数）
用户：确认 / 给出修改意见
AI：修改第 1 幕（如有反馈）→ 生成第 2 幕 → 展示（附累计字数）
...重复直到本章 3～5 幕全部完成，最终控制在 2000～4000 字
AI：自动执行章级检查
```

规则：
- 每幕生成后暂停，等待用户确认
- 用户说"继续"确认当前幕，也可给出修改意见
- 修改意见可以是具体改动，也可以是方向性调整
- 每幕生成后 AI 告知当前字数，如偏离幕结构建议的预估字数过大，提示作者是否需要调整后续幕的篇幅
- 第 N 幕起始必须承接第 N-1 幕的落点，不能无说明地跳跃视角或时间
- 最后一幕必须落到章末钩子
- 全章总字数控制在 2000～4000 字
- 全部幕完成后自动触发章级检查

## 十五、设定变更的级联更新

当检查发现设定冲突时：

```
确认冲突类型
  ├─ 正文错误 → 修改正文 → 重新检查
  └─ 设定需更新 → 执行以下级联流程
```

**级联流程**：

```
1. 修改设定文件（如 settings/power-system.md）
     ↓
2. 读该文件底部的「反向链接」段，获取所有引用者
   例如：[[protagonist]]、[[antagonist]]、[[mentor]]
     ↓
3. 逐一检查引用者文件中与变更内容相关的部分
   例如：power-system 中"筑基期"定义改了
        → 检查 protagonist.md 能力演变表中是否引用了旧定义
        → 检查 antagonist.md 是否有相关内容需要同步
     ↓
4. 更新受影响的引用者文件
     ↓
5. 如引用关系发生变化（新增/移除链接），刷新相关文件的反向链接
     ↓
6. 重新检查
```

如果反向链接为空或缺失（如刚开书时），回退为全文搜索 `[[被修改的设定文件名]]`。但正常情况下，反向链接应该在 `/拆大纲` 时初始化完毕。

## 十六、技术实现方式

作为 Claude Code 的 custom slash command（`.claude/commands/` 下的 md 文件）实现，不依赖外部 Python 脚本。

核心 command 文件：
- `.claude/commands/开书.md`
- `.claude/commands/定设定.md`
- `.claude/commands/拆大纲.md`
- `.claude/commands/写章.md`
- `.claude/commands/检查.md`
- `.claude/commands/更新设定.md`
- `.claude/commands/伏笔.md`
- `.claude/commands/角色状态.md`
- `.claude/commands/风格审查.md`

每个 command 文件包含：命令描述和参数说明、详细执行步骤、输出格式规范、禁止事项。

`templates/` 目录存放各文件的初始模板，开书时从模板生成项目文件。

## 十七、验证方式

1. 用一个短篇（3 章）跑通完整工作流：开书 → 定设定 → 拆大纲（含弧光规划）→ 逐幕写章 → 章级检查

3. 验证角色弧光规划 → 角色演变追踪 的对照链路是否有效
4. 验证伏笔规划总表 → foreshadow.md 状态追踪 的同步链路
5. 验证 Obsidian 链接的按需加载策略是否有效控制上下文消耗
